# Product Requirements Prompt (PRP) — ✅ Completed

## 1. Project Context & Objectives
The `NotationCanvas` component in the Chord Notator plugin suffers from a state desynchronization bug during chord transformations. When notes are transformed via UI or hardware bindings (e.g., Chromatic Shift, Diatonic Shift, PCS Rotation), the `note` property of the `ActiveNoteData` object is updated, but its `sourceMidi` property retains the original hardware input value. This orphaned `sourceMidi` value blocks the virtual/physical keyboard from re-adding the original pitch to the new chord, as the input validator incorrectly flags the original pitch as already active. The objective is to patch the transformation functions to update `sourceMidi` synchronously with the `note` property.

## 2. Technical Decisions & Dependencies
* **Target File:** `src/plugins/chord-notator/components/NotationCanvas.tsx`
* **Affected Functions:** `applyChromaticShift`, `applyDiatonicShift`, `applyPcsRotation`
* **Logic Specification:** Within the mapping functions that generate `updatedNotes`, explicitly overwrite `sourceMidi` with the newly calculated `safePitches[index]` or `note` value. This permanently decouples the transformed note from its original physical/virtual trigger, allowing the original trigger to be struck again.

## 3. Implementation Phases

### Phase 1: Patch Transformation Functions — ✅ Completed
* **Objective:** Ensure `sourceMidi` tracks the new pitch after any transformation.
* **Instructions:**
