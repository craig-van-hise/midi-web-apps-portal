# Product Requirements Prompt (PRP) — Sequence Copy Freeze Fix ✅

## 1. Project Context & Objectives
The application is experiencing a total UI freeze (infinite loop) when a user copies a chord in the `StepSequencer` and subsequently attempts to transpose it. This is caused by an object spread error during the copy event where raw primitive numbers are incorrectly destructured into empty objects, causing `NaN` values to propagate into the transposition and layout engines. We need to fix the copy data handling and add a strict iteration limit (fail-safe) to the layout engine's `while` loops.

## 2. Technical Decisions & Dependencies
* **Target Files:** `src/plugins/chord-notator/components/StepSequencer.tsx` (and optionally `NotationCanvas.tsx` if it shares the layout logic).
* **Root Cause:** Spreading a number primitive `const obj = { ...60 }` results in `{}`. 
* **Fail-Safe Mechanism:** All `while (!placed)` loops in the layout engine computing accidental or column offsets must have a hard breakout condition (e.g., `if (col > 20) break;`) to guarantee the main thread can never be blocked by corrupt data.

## 3. Implementation Phases

### Phase 1: Fix Note Object Cloning (`StepSequencer.tsx`) — ✅ Completed
* **Objective:** Ensure the Option+Drag copy logic correctly handles both raw numbers and populated note objects.
* **Instructions:**
  1. Locate the `onPointerUp` drop handler inside the `StepSequencer` chord pill rendering (around line 282).
  2. Replace the faulty `copiedNotes` mapping:
     ```javascript
     // BAD
     const copiedNotes = sourceNotes.map(n => ({
       ...n,
       id: generateId()
     }));
     ```
  3. With a type-safe check:
     ```javascript
     // GOOD
     const copiedNotes = sourceNotes.map(n => {
       if (typeof n === 'object' && n !== null) {
         return { ...n, id: generateId() };
       }
       return n; // If it's a raw number, leave it alone.
     });
     ```

### Phase 2: Layout Engine Loop Fail-Safes (`StepSequencer.tsx`) — ✅ Completed
* **Objective:** Prevent the layout engine from infinitely looping if it receives corrupt `NaN` step values.
* **Instructions:**
  1. Locate the `computeMiniLayout` function at the top of `StepSequencer.tsx`.
  2. Find the `while (!placed)` loops (there should be one for note columns and one for accidental columns).
  3. Inject a safety breakout constraint:
     ```javascript
     let col = 0;
     let placed = false;
     while (!placed && col < 20) { // HARD LIMIT PREVENTS HANGS
       if (!columns[col]) columns[col] = [];
       // If finalStep is NaN, fallback to 0 to prevent logic collapse
       const safeStep = isNaN(note.finalStep) ? 0 : note.finalStep; 
       
       if (!columns[col].some(existingStep => Math.abs(existingStep - safeStep) <= 3)) {
         columns[col].push(safeStep);
         // ... rest of logic
         placed = true;
       } else { 
         col++; 
       }
     }
     ```
