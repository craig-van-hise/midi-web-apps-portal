# Product Requirements Prompt (PRP) — 🔵 Active

## 1. Project Context & Objectives
**Context:** In the `chord-notator` plugin, the recent update to the transformations drawer's closed state resulted in incorrect positioning. The main body of the drawer is still partially visible, and there is a visible gap between the keyboard and the drawer.
**Objective:** Correct the layout math and stacking context so that when the drawer is in the "closed" state, the main body of the drawer is completely hidden behind or flush against the bottom of the keyboard container. **Only** the chevron tab should be visible, protruding directly from the bottom edge of the keyboard card.

## 2. Technical Decisions & Dependencies
* **Layout Strategy (The "Flush" Requirement):** * The transformations drawer must sit on a lower `z-index` than the main keyboard container so it can cleanly slide beneath it.
  * The chevron toggle button should be absolutely positioned to hang off the bottom edge of the drawer container (e.g., `bottom: -[tab_height]`, or styled as an appended element).
  * The closed state animation (via Framer Motion `variants` or CSS `transform: translateY`) must shift the drawer exactly by the height of its *main body*. If using Framer Motion, the closed state should likely be `y: "-100%"` (relative to its own container) or exactly `y: 0` if it is positioned strictly behind the keyboard, depending on the current DOM structure.
* **Tolerance:** There must be exactly `0px` of white space/gap between the bottom of the keyboard card and the top of the visible tab. 

## 3. Implementation Phases

### Phase 1: DOM Structure & Z-Index Audit — ✅ Completed
* **Objective:** Ensure the HTML structure and stacking context support a flush hide.
* **Instructions:**
  1. Inspect the wrapper containing both the Keyboard and the Drawer (e.g., `Keyboard.tsx` or `NotationCanvas.tsx`).
  2. Ensure the Keyboard component has a higher `z-index` and a solid background to obscure elements behind it.
  3. Ensure the Drawer component has a lower `z-index` so it cleanly tucks under the keyboard.
  4. Verify that the chevron tab is structurally extending below the main background of the drawer.
* **TDD Checkpoint:**
  * **Test Case 1:** *Given* the elements render, *Then* the keyboard's bounding box completely overlaps the drawer's bounding box when the drawer is translated up.
  * **Status Update:** "Transition Phase status to 🔵 Active. Execute structural CSS fixes. Once complete, update status to ✅ Completed."

### Phase 2: Precise Position Calculation — ✅ Completed
* **Objective:** Adjust the closed state offsets to hide 100% of the drawer body.
* **Instructions:**
  1. Locate the Framer Motion `variants` or CSS classes controlling the open/closed states.
  2. Modify the closed state `translateY` (or `y`) value. Instead of guessing the offset, calculate it precisely. If the tab hangs completely outside the drawer's main wrapper, the closed state `y` offset relative to the keyboard should hide the wrapper entirely. 
  3. Remove any margins or gaps (e.g., `gap-4`, `mt-2`) between the keyboard flex container and the drawer container that might be causing the white space seen in the error screenshot.
* **TDD Checkpoint:**
  * **Test Case 1:** *Given* the drawer is closed, *Then* the calculated pixel/percentage offset exposes the chevron tab and 0px of the drawer's main padding/background.
  * **Status Update:** "Transition Phase status to 🔵 Active. Apply precise layout calculations. Once applied, update status to ✅ Completed."
