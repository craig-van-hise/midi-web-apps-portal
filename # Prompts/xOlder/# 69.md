# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
**Context:** The `chord-notator` plugin in the `midi-web-apps-portal` is experiencing a state synchronization bug. 
**Bug Description:** After a user inputs a chord, applies a "rotate" transformation, and subsequently removes notes (via the UI keyboard or notation canvas), the displayed chord symbol fails to update. It remains locked on the chord symbol calculated immediately after the rotation.
**Objective:** Identify the state disconnect between the transformation logic, note removal logic, and the `chordSpeller` utility, ensuring the chord symbol always reactively reflects the current pitch set.

## 2. Technical Decisions & Dependencies
* **Root Cause Hypothesis:** The "rotate" transformation is likely mutating a secondary state (or an orphaned array reference) rather than the primary `activeNotes` (or `selectedNotes`) single source of truth, OR a derived state hook (`useMemo`/`useEffect`) responsible for calling `chordSpeller.ts` is missing a dependency related to the transformed array.
* **Architecture Standard:** The application uses throttled `useRef` updates for performance (as per `PROJECT_STATE.md`). The fix must maintain this `useRef` + `lodash/throttle` pattern without introducing synchronous main-thread blocking during MIDI input.
* **Target Files:** * `src/plugins/chord-notator/hooks/` (State management hooks)
  * `src/plugins/chord-notator/components/NotationCanvas.tsx` / `Keyboard.tsx`
  * `src/plugins/chord-notator/components/NotationCanvas.bugs.test.tsx` (For reproducing the specific bug)
