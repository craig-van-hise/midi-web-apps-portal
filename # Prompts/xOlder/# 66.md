# Product Requirements Prompt (PRP) — 🔘

## 1. Project Context & Objectives
The `chord-notator` module within the `midi-web-apps-portal` currently relies on a root-based spelling algorithm paired with `chord_intervals` from a `PCS_LUT` lookup. While functional for isolated chords, this approach produces suboptimal, non-musical notation (e.g., spelling a cluster as B, C, Cx, D#) when the played Pitch Class Set (PCS) is a subset of the currently selected key or scale (e.g., C Melodic Minor). 

**Objective:** Implement a contextual spelling override algorithm. The system must evaluate if the incoming active MIDI notes form a pitch-class subset of the globally selected Root + Scale. If true, it must enforce the diatonic spellings defined by that scale, bypassing the generic `PCS_LUT` interval logic.

## 2. Technical Decisions & Dependencies
* **Target Files:** The logic will primarily reside in `src/plugins/chord-notator/utils/chordSpeller.ts` (and potentially `notationMath.ts`).
* **Algorithm Strategy:**
    1.  **Scale Map Generation:** Create a deterministic mapping of Pitch Classes (0-11) to their strict string spellings based on the active `Key` and `ScaleType`. For example, C Melodic Minor maps `{0:'C', 2:'D', 3:'Eb', 5:'F', 7:'G', 9:'A', 11:'B'}`.
    2.  **Subset Validation:** On incoming MIDI evaluation, extract the unique pitch classes (`note % 12`) from the active cluster. Perform a strict subset mathematical check against the active Scale Map's keys.
    3.  **Override Routing:** If `isSubset === true`, map the MIDI notes to their string representations using the Scale Map and append the correct octave (`Math.floor(note / 12) - 1`). If `false`, fall back to the existing `PCS_LUT` root/interval algorithm.
* **Constraints:** Do not introduce heavy external music theory libraries (like `tonal.js`) if internal `utils` can handle the mapping. Keep operations synchronous and highly optimized for the main audio thread.
