
> **CRITICAL ARCHITECTURE PIVOT: The "Plugin Host" Monolith**
> **Context:** We are abandoning the `iframe` and `window.postMessage` architecture. Managing multiple local dev servers has become unscalable. We are converting the `midi-web-apps-portal` into a single unified React application using an "Internal Plugin" structure.
> **The Goal:** The Portal will act as the master "DAW" harness. The individual MIDI web apps will be brought inside the repository as completely isolated React components (Plugins).
> **Step 1: Restructure the `src` Directory (Strict Siloing)**
> Create the following strict directory structure inside the `midi-web-apps-portal` repo:
> 1. `src/core/` -> Move all existing Portal harness UI here (Sidebar, TopBar, Rompler, App layout).
> 2. `src/plugins/` -> This is the new home for the individual apps.
> 
> 
> **Step 2: Porting the Apps into Plugins**
> Create a subfolder inside `src/plugins/` for each app (e.g., `src/plugins/chord-notator`, `src/plugins/pitch-class-matrix`).
> You must port the core UI and logic from the adjacent repositories into these subfolders.
> *CRITICAL RULE:* These plugins must remain strictly isolated. They cannot share utility files or components with each other. They must export a single, default React component from an `index.jsx` file at the root of their respective folder.
> **Step 3: Eradicating the Iframe & Message Bus**
> Remove the `<iframe>` from the Portal's Main Pane.
> Remove the `useMessageBus` and `usePortalLink` files entirely.
> The Portal will now render the active app directly as a standard React component.
> **Step 4: The New Communication Standard (Props)**
> The Portal still owns the `navigator.requestMIDIAccess()` and the Tone.js Rompler.
> When the Portal renders the active plugin, it must pass data down via props, and receive data up via callbacks.
> Example implementation for the Main Pane:
> ```jsx
> const ActivePlugin = getPluginComponent(activeAppId); // dynamically grab from registry
> 
> return (
>   <main className="main-pane">
>     <ActivePlugin 
>       midiIn={globalMidiData} // Raw hardware MIDI passed down
>       onMidiOut={handleRomplerOutput} // Processed MIDI sent up to Rompler
>       isBypassed={isPowerOff}
>       showInfo={isInfoModalOpen}
>       showSettings={isSettingsModalOpen}
>       triggerPanic={panicTriggerState}
>     />
>   </main>
> );
> 
> ```
> 
> 
> **Execution:** Start by establishing the `src/core/` and `src/plugins/` folder structures, rewriting the Portal's Main Pane to render a dummy plugin via Props instead of an iframe, and run tests. Do not attempt to port all 5 apps at once. Start with the `chord-notator` as the proof-of-concept plugin. Report back when the harness is ready to test the first internal plugin.
