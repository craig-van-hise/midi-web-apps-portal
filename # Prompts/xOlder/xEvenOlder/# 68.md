
# Product Requirements Prompt (PRP) — 🔘

## 1. Project Context & Objectives

In the `chord-notator` plugin, utilizing the `SEMI` (transpose) controls on a rotated (locked) chord results in a delayed UI render. The displayed chord name becomes stale and only updates when a subsequent mouse movement forces a React render cycle. Furthermore, when the delayed render occurs, the identity lock is incorrectly broken or miscalculated, resulting in an entirely new chord identity instead of mathematically transposing the locked identity.

**Objective:** Fix the React rendering lifecycle for `SEMI` and `ROT` actions to ensure immediate, synchronous UI updates. Debug and patch the `chordIdentityRef` transposition logic to ensure the lock remains unbroken and accurately reflects the transposed `baseName` and `rootPC`.

## 2. Technical Decisions & Dependencies

* **Target Files:** The component managing the state/hooks for the Notator (likely `NotationCanvas.tsx` or `useWebMidi.ts`/`index.jsx`).
* **Root Cause 1 (Stale UI):** The transformation handlers (`handleSemiUp`, `handleRotDown`, etc.) are mutating a `useRef` or internal array but failing to invoke the `syncNotesUI()` throttled function (or equivalent React state setter).
* **Root Cause 2 (Lost Lock):** The `useEffect` listening to the `midiBus` might be accidentally catching a UI-triggered internal MIDI routing event and treating it as fresh hardware input, thereby setting `chordIdentity.isActive = false`. Alternatively, the transposition logic is failing to accurately update the `baseName` string (e.g., shifting "Gø9" to "Abø9").

## 3. Implementation Phases

### Phase 1: Force UI Render on Transformation — 🔵 Active

* **Objective:** Ensure that clicking `SEMI` or `ROT` instantly triggers a React state update so the user does not have to move the mouse to see the new notes/name.
* **Instructions:**
1. Locate the event handler functions for the `SEMI` and `ROT` UI buttons.
2. At the end of the logic block inside these handlers (after the active notes and `chordIdentityRef` have been mathematically updated), explicitly call the React state setter for the active notes (or invoke the `syncNotesUI()` function if using the throttle pattern).
3. Ensure no local state updates are wrapped in asynchronous callbacks that might get swallowed.


* **TDD Checkpoint:** * **User Verification:** "[USER VERIFICATION REQUIRED] Rotate a chord. Click the SEMI up arrow. Take your hand completely off the mouse. Does the UI update instantly to show the shifted notes and new chord name?"
* **Status Update:** "Transition Phase status to 🔵 Active. Perform user verification. Once tests pass successfully, update status to ✅ Completed before initiating subsequent phases."



### Phase 2: Patch the Transposition Naming Logic — 🔘

* **Objective:** Ensure that when `SEMI` is clicked while a lock is active, the `baseName` updates correctly and the lock is preserved.
* **Instructions:**
1. Review the `SEMI` handler block where `chordIdentityRef` is transposed.
2. Verify the root math: `const newRootPC = (chordIdentityRef.current.rootPC + delta + 12) % 12;`
3. **Crucial Fix:** You must map `newRootPC` to its string equivalent (using your key/scale spelling map or a generic chromatic map) and replace the root letter in `chordIdentityRef.current.baseName`. (e.g., If `baseName` is "Cmaj7" and delta is +1, you must slice out "C" and inject "Db" or "C#" to make "Dbmaj7").
4. Ensure `chordIdentityRef.current.isActive = true` remains explicitly set throughout this process.


* **TDD Checkpoint:** * **Test Case 1:** [Given `baseName` is "Gø9" and `rootPC` is 7. When `SEMI` delta is +1. Assert `rootPC` becomes 8 and `baseName` becomes "Abø9" (or G#ø9 depending on accidental preference)].
* **Status Update:** "Transition Phase status to 🔵 Active. Author and run these verification tests. Once tests pass successfully, update status to ✅ Completed before initiating subsequent phases."



### Phase 3: Protect the Lock from Internal Events — 🔘

* **Objective:** Prevent the delayed render or UI interaction from accidentally breaking the identity lock.
* **Instructions:**
1. Inspect the `useEffect` that listens for `midiBus` hardware input (the one responsible for setting `chordIdentity.isActive = false`).
2. Ensure that UI-generated transformations (which might be routing through `onMidiOut` or an internal event bus to generate audio) do not accidentally loop back into the hardware `midiBus` listener.
3. If necessary, tag UI-generated notes with a specific flag (e.g., `isSynthetic: true`) so the hardware listener can ignore them and preserve the lock.


* **TDD Checkpoint:** * **Test Case 1:** [Given a locked chord identity. When an internal synthetic MIDI routing event fires. Assert `chordIdentity.isActive` remains `true`].
* **Status Update:** "Transition Phase status to 🔵 Active. Author and run these verification tests. Once tests pass successfully, update status to ✅ Completed before initiating subsequent phases."



## 4. Final Review & Validation

* **Technical Audit:** Verify that rapid clicking of the `SEMI` arrows does not cause UI stuttering or lag, and that the names transpose flawlessly without reverting to raw `PCS_LUT` output.
* **Status Reconciliation:** Verify all sub-phases match ✅ or ❌ before updating top-level status to ✅.
