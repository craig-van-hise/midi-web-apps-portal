# Product Requirements Prompt (PRP) — 🔵 Active

## 1. Project Context & Objectives
**Context:** The transformations drawer in the `chord-notator` is completely broken visually. 1) In the open state, it is floating in space due to an unwanted margin or flex gap. 2) In the closed state, its z-index stacking context is wrong—it is sliding behind the piano keys, but in front of the keyboard card's background, creating a visual disaster.
**Objective:** Establish a strict, bulletproof DOM hierarchy and CSS stacking context. The keyboard card must act as an opaque ceiling (`z-10`). The drawer must sit perfectly flush against the bottom of the keyboard card (0px gap), and slide cleanly behind the *entire* keyboard card wrapper (`-z-10`) when closed.

## 2. Technical Decisions & Dependencies
* **The Gap Issue:** The parent container holding both the Keyboard and the Drawer likely has a `gap` class (e.g., `gap-4`), or the Drawer has a `margin-top`. This MUST be removed so the Drawer touches the Keyboard Card.
* **The Stacking Context Issue:** The Drawer is slipping between layers. To fix this, the main wrapper of the Keyboard Card MUST have a solid background color (e.g., `bg-white` or dark mode equivalent), `relative` positioning, and a higher `z-index` (e.g., `z-10`). 
* **The Drawer Context:** The Drawer MUST be a sibling to the Keyboard Card, not nested inside the key rendering area. It must have a lower z-index (e.g., `-z-10`).

## 3. Implementation Phases

### Phase 1: Parent Layout & Gap Removal — ✅ Completed
* **Objective:** Remove the floating gap so the open drawer sits perfectly flush against the keyboard.
* **Instructions:**
  1. Inspect the parent container wrapping both the Keyboard Card and the Drawer.
  2. Remove any flexbox gaps (e.g., `gap-4`, `gap-y-4`) that force space between the two sibling elements.
  3. Inspect the Drawer component wrapper and remove any top margins (`mt-2`, `mt-4`, etc.).
* **TDD Checkpoint:**
  * **Test Case 1:** *Given* the drawer is open, *Then* there are exactly 0 pixels of space between the bottom border of the keyboard card and the top border of the drawer.
  * **Status Update:** "Transition Phase status to 🔵 Active. Strip the margins/gaps. Once applied, update status to ✅ Completed."

### Phase 2: Stacking Context Lockdown — ✅ Completed
* **Objective:** Force the Keyboard Card to be an opaque ceiling so the drawer cannot slip between the keys and the background.
* **Instructions:**
  1. Apply the following CSS/Tailwind rules to the outermost wrapper of the **Keyboard Card**: `relative z-10`. Ensure it has a solid background color (not transparent) so elements behind it cannot be seen.
  2. Ensure the wrapper for the **Drawer Component** is a direct sibling to the Keyboard Card wrapper.
  3. Apply `relative -z-10` to the Drawer's outermost wrapper.
* **TDD Checkpoint:**
  * **Test Case 1:** *Given* the drawer translates upwards, *Then* it disappears completely behind the Keyboard Card's solid background, entirely invisible except for the tab.
  * **Status Update:** "Transition Phase status to 🔵 Active. Lock down the z-indices and DOM hierarchy. Once applied, update status to ✅ Completed."

### Phase 3: Slide-Behind Translation & Tab Anchoring — 🔵 Active
* **Objective:** Ensure the closed state math is correct and only exposes the tab.
* **Instructions:**
  1. With the flex gap removed, the open drawer should sit perfectly below the keyboard.
  2. The tab must be appended to the bottom of the drawer's card.
  3. The Framer Motion `closed` variant for the drawer wrapper should use the negative translation: `y: "calc(-100% + [height_of_tab]px)"`. (e.g., if the tab is 24px tall, `y: "calc(-100% + 24px)"`).
* **TDD Checkpoint:**
  * **User Verification (Required):** "[USER VERIFICATION REQUIRED] Pause for user review. Please run `npm run dev`. Confirm that: 1) The open state sits flush with no floating gap. 2) The closed state hides the card completely behind the solid white keyboard background (NOT clipping through the keys). 3) Only the tab is visible. Reply 'Approved' when confirmed."
  * **Status Update:** "Transition Phase status to 🔵 Active. Await user confirmation. Once approved, update status to ✅ Completed."

## 4. Final Review & Validation
* **Technical Audit:** Ensure that `-z-10` on the drawer doesn't make its buttons unclickable when open. If the parent container needs a specific stacking context to allow clicks, adjust accordingly (e.g., parent is `isolate`, drawer is `z-0`, keyboard is `z-10`).
* **Status Reconciliation:** Verify all sub-phases match ✅ Completed before updating the top-level status to ✅ Completed.
* **Final Sign-off:** Present the exact CSS classes and structure to the user for final closure.
