# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
The VV | WebApps Portal is experiencing critical polyphonic MIDI degradation (random hung notes) when processing chords. Analysis of the provided source files reveals two primary architectural failures:
1.  **React 18 State Batching (The Root Cause):** The Web MIDI API fires rapid, sequential `midimessage` events during polyphonic chord strikes and releases. In `App.jsx`, `setGlobalMidiData(data)` overwrites the state synchronously before React can render. Consequently, plugins only receive the *last* message of the batch (dropping crucial Note Offs for the preceding notes in the chord).
2.  **AudioEngine Orphaned Nodes:** `Tone.Sampler` and `smplr` do not possess explicit duplicate-trigger guards. If a race condition allows a duplicate Note On before a Note Off, an orphaned audio node is created, which `triggerRelease` cannot garbage collect.

**Objective:** Refactor the host-to-plugin data pipeline to use a batched message queue, completely bypassing React's single-value state overwrites. Concurrently, implement a strict `activeVoices` Set in the `AudioEngine` to prevent ghost node generation.

## 2. Technical Decisions & Dependencies
* **Message Queue Pattern:** To maintain the README-mandated "prop-based architecture" while defeating state batching, `globalMidiData` must be transformed from a singular array into an accumulated queue of timestamped event objects.
* **Data Structure:** `MidiEvent = { id: string, data: number[], timestamp: number }`.
* **Voice Allocation Guard:** `engine.js` will utilize a native Javascript `Set<string>` to track active polyphonic voices and explicitly block duplicate `noteOn` allocations for an already active key.

### Phase 1: Implement Global MIDI Queue in `App.jsx` — 🔘 Pending
* **Objective:** Prevent React from overwriting rapid polyphonic MIDI messages by accumulating them in an array queue.
* **Instructions:**
    1.  In `src/core/App.jsx`, replace `const [globalMidiData, setGlobalMidiData] = useState(null);` with `const [midiQueue, setMidiQueue] = useState([]);`.
    2.  Update the `handleMidiMessage` listener inside the Web MIDI `useEffect`:
        * Instead of replacing state, append to it: `setMidiQueue(prev => [...prev, { id: crypto.randomUUID(), data: Array.from(message.data), ts: performance.now() }]);`

        * `const consumeMidiQueue = useCallback((ids) => { setMidiQueue(prev => prev.filter(msg => !ids.includes(msg.id))); }, []);`

          if (midiInQueue && midiInQueue.length > 0 && !isBypassed) {

              processMidi(msg.data);
              idsProcessed.push(msg.id);

### Phase 4: Harden AudioEngine Voice Allocation (`engine.js`) — 🔘 Pending
* **Objective:** Prevent `Tone.Sampler` from generating orphaned internal oscillator nodes during rapid identical note strikes.
* **Instructions:**
    1.  In `src/core/rompler/engine.js`, add `this.activeVoices = new Set();` inside the `AudioEngine` constructor.
    2.  Modify the `noteOn(note, velocity = 1)` method:
        * Before triggering the sampler, check: `if (this.activeVoices.has(note)) return;`
        * If it proceeds: `this.activeVoices.add(note);`
    3.  Modify the `releaseNote(note)` method:
        * At the start, check: `if (!this.activeVoices.has(note)) return;`
        * If it proceeds: `this.activeVoices.delete(note);`
    4.  Modify `releaseAll()`:
        * Clear the set: `this.activeVoices.clear();`
