# Product Requirements Prompt (PRP) — 🔵 Active

## 1. Project Context & Objectives
**Context:** The previous attempt to fix the closed state of the transformations drawer using an `overflow: hidden` / `height: 0` wrapper severely broke the open state. It introduced unwanted margins, ruined the drop shadow continuity, and awkwardly detached the tab from the main card.
**Objective:** 1. **Revert:** Immediately strip out the height-collapse wrapper and restore the exact layout and DOM structure of the original open state (flush against the keyboard, cohesive shadow, tab physically attached). 
2. **Fix:** Re-implement the `translateY` (slide-behind) strategy, but correctly anchor the drawer to the bottom of the keyboard and apply the exact offset needed so the main body tucks behind the keyboard *without* overshooting the top boundary.

## 2. Technical Decisions & Dependencies
* **The Root Cause of the "Peek":** In the earlier version, the drawer peaked out the top of the keyboard when closed because its closed-state `y` offset was calculated poorly or it was anchored incorrectly.
* **The Math Fix:** - The entire drawer component (Main Card + Tab) must be absolutely positioned and anchored to the *bottom* of the keyboard container (e.g., `top: 100%`), with a negative `z-index` so it sits behind it.
  - The closed state animation should translate the drawer up by exactly the height of the Main Card. Because the Main Card is significantly shorter than the keyboard, sliding it up by its own height will completely hide it behind the keyboard *without* it reaching the top edge.
  - Example: If the tab is `32px` tall, the closed variant is `y: "calc(-100% + 32px)"`.

## 3. Implementation Phases

### Phase 1: Revert and Restore Open State — ✅ Completed
* **Objective:** Undo the broken wrapper and restore the perfect open state UI.
* **Instructions:**
  1. Remove the `overflow: hidden` wrapper and the `height: auto`/`height: 0` Framer Motion variants.
  2. Reattach the chevron tab to the main drawer card so they act as a single visual unit with a shared or cohesive drop shadow.
  3. Ensure the drawer sits perfectly flush against the bottom of the keyboard in its open state (no gaps, no weird margins).
* **TDD Checkpoint:**
  * **Test Case 1:** *Given* the drawer is set to open, *Then* the UI renders identically to the original intended design, with the tab attached to the card and zero gap between the drawer and the keyboard.
  * **Status Update:** "Transition Phase status to 🔵 Active. Execute the revert. Once applied, update status to ✅ Completed."

### Phase 2: Precise Slide-Behind Math — ✅ Completed
* **Objective:** Hide the drawer without overshooting the keyboard's top edge.
* **Instructions:**
  1. Wrap the keyboard container in a `relative` div with a solid background and a high `z-index`.
  2. Anchor the drawer wrapper just below the keyboard: `absolute left-0 right-0 top-full -z-10`.
  3. Update the Framer Motion `variants` on the drawer:
     * `open: { y: 0 }`
     * `closed: { y: "calc(-100% + [exact-height-of-tab])" }` (e.g., `calc(-100% + 24px)`). This slides the entire unit up just enough to hide the main body, leaving only the tab exposed.
* **TDD Checkpoint:**
  * **Test Case 1:** *Given* the drawer is closed, *Then* the component translates up. The main body is completely hidden behind the keyboard, the top of the drawer does NOT breach the top of the keyboard, and the tab remains visible.
  * **Status Update:** "Transition Phase status to 🔵 Active. Implement the corrected transform math. Once applied, update status to ✅ Completed."
