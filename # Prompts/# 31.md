# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
The **VV | WebApps Portal** is a unified, browser-based MIDI processing environment currently suffering from latency bottlenecks due to standard Web Audio API heuristics, JavaScript garbage collection, and main-thread blocking. The objective of this PRP is to systematically overhaul the audio architecture to achieve professional, sample-accurate, sub-5ms latency. This will be accomplished by forcing interactive latency hints, neutralizing garbage collection via object pooling, and migrating the MIDI-to-Audio bridge to an `AudioWorklet` utilizing a lock-free `SharedArrayBuffer`.

## 2. Technical Decisions & Dependencies
To accomplish the low-latency mandate, you must adhere to the following architectural decisions:
* **Cross-Origin Isolation:** `SharedArrayBuffer` is disabled in modern browsers unless the document is cross-origin isolated. You **MUST** update `vite.config.js` to serve the application with `Cross-Origin-Opener-Policy: same-origin` and `Cross-Origin-Embedder-Policy: require-corp` headers.
* **AudioContext Initialization:** Tone.js must be explicitly initialized with a custom context using `latencyHint: "interactive"`. 
* **Ring Buffer Algorithm:** The `SharedArrayBuffer` bridge will use a fixed-size `Uint8Array` (e.g., 1024 bytes) as a Ring Buffer. You will need a `writePointer` (managed by the Main Thread) and a `readPointer` (managed by the AudioWorklet), using `Atomics` to prevent race conditions without locking.
* **Voice Pooling:** Instead of dynamic instantiation inside the MIDI event listener, allocate an array of exactly 128 voice objects (to match the MIDI specification maximum polyphony) at startup.
