# Product Requirements Prompt (PRP) — 🔘 Pending

## 1. Project Context & Objectives
The VV | WebApps Portal's MIDI input stage has been successfully refactored to a synchronous Event Bus, resolving polyphonic hung notes. However, a baseline latency of ~15-20ms remains. This latency is suspected to be introduced at the Web Audio API / Tone.js scheduling layer, not the JavaScript plugin execution layer.

**Objectives:**
1. Implement a passive, non-destructive Latency Profiler to calculate the JS execution delta and Web Audio buffer delays.
2. Optimize `engine.js` Web Audio context initialization for real-time interactive performance.

## 2. Technical Decisions & Dependencies
* **Latency Profiler:** A simple Singleton or global utility that hooks into `handleMidiMessage` (Start) and `audioEngine.noteOn` (End). It will calculate:
  - `JS Delta`: Execution time through plugins.
  - `Output Latency`: `Tone.context.outputLatency` (inherent OS/Browser audio delay).
* **Audio Engine Optimization:** - Change `Tone.context.lookAhead` from `0.01` to `0` or `0.005`.
  - Ensure the raw AudioContext is instantiated with `{ latencyHint: "interactive" }`.

## 3. Implementation Phases

### Phase 1: Implement the Passive Latency Profiler — 🔘 Pending
* **Objective:** Create a diagnostic utility to measure processing time without mutating any existing plugin logic.
* **Instructions:**
    1. Create a new utility file: `src/core/utils/latencyProfiler.js`.
    2. Implement a simple class or object that stores a `Map` of MIDI note numbers to `performance.now()` timestamps.
    3. Expose two methods: `markInput(note)` and `markAudioTrigger(note)`.
    4. In `markAudioTrigger`, calculate the delta: `const jsDelta = performance.now() - startTime`. 
    5. Log the `jsDelta` alongside `Tone.context.outputLatency * 1000` (to convert seconds to ms) in the console as a warning or info log, prefixed with `[VV Profiler]`.
    6. In `App.jsx`, inside `handleMidiMessage`, call `latencyProfiler.markInput(note)` for Note Ons.
    7. In `engine.js`, inside `noteOn()`, call `latencyProfiler.markAudioTrigger(note)` right before the sampler triggers.
* **TDD Checkpoint:**

### Phase 2: Tone.js Real-Time Optimization — 🔘 Pending
* **Objective:** Reduce Web Audio buffer sizes and look-ahead times.
* **Instructions:**
    1. In `src/core/rompler/engine.js`, locate the `init()` method.
    2. Currently, Tone.js might be using the default AudioContext. Force a low-latency context *before* `Tone.start()` if possible, or adjust the global Tone settings.
    3. Update the lookAhead: Change `Tone.context.lookAhead = 0.01;` to `Tone.context.lookAhead = 0;` (or `0.005` if 0 causes clipping/audio artifacts).
    4. Ensure the Web Audio API context is interactive:
       ```javascript
       if (Tone.context.rawContext && Tone.context.rawContext.baseLatency !== undefined) {
         // Tone.js automatically attempts interactive latency, but we force lookAhead down.
         Tone.context.lookAhead = 0.002; // 2ms safety buffer instead of 10ms
       }
       ```
* **TDD Checkpoint:**
